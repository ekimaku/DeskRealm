# DeskRealm v0.5.9

DeskRealm `v0.5.9` is a UX release. It keeps the safe `v0.5.8` first-run model, where the original Windows Desktop is associated without moving files, and adds the user-facing window needed to make DeskRealm easier to understand, configure and control. It also adds the DeskRealm `DR` logo to the compiled executable, tray notification icon and main window.

## What changed since v0.5.8

### First-run onboarding

Fresh installs now open the DeskRealm window before the first automatic Desktop switch. The onboarding explains what DeskRealm does and lets the user either associate the original Windows Desktop with a realm, or skip association and create visible shortcuts back to the original Desktop.

### Main DeskRealm UI

The new tray-opened UI exposes onboarding, hotkeys, Icon Layout management, actions, options and status. Normal usage stays tray-first: closing the window hides it to the tray, while **Quit DeskRealm** remains the explicit app exit.

The UI was also visually polished with a modern dark/cyan shell, custom navigation, rounded cards, custom buttons, readable lock states and dark in-app chrome.

### Hotkey capture

Hotkeys can now be captured directly from the UI. The field records a shortcut when the first non-modifier key is pressed. Modifier-only attempts cancel when the modifiers are released. Existing custom hotkeys are preserved during migration.

Default hotkeys changed to avoid `Win+Shift+W` and `Win+Shift+V`:

- Desktop 1: `Win+Shift+X`
- Desktop 2: `Win+Shift+C`
- Desktop 3: `Win+Shift+B`
- Desktop 4: `Win+Shift+N`

### Icon Layout management

The new **Icon Layout** tab shows realms as parent rows and saved display-topology variants as child rows. Each variant row shows the persisted per-monitor working resolutions and marks the primary display with `✅`.

Users can now:

- lock a layout;
- lock a realm, protecting every child variant assigned to that realm;
- lock one exact display-topology variant;
- manually overwrite a locked layout only after confirmation;
- delete stale saved variants after confirmation.

### Pause semantics

The old ambiguous enabled checkbox is now **Enable realm switching automation**. When disabled, automatic realm switching and DeskRealm desktop hotkeys are paused without deleting assignments, realms, files, icons or layouts.

## Safety notes

- First-run onboarding does not move Desktop files.
- The first automatic Desktop switch waits until onboarding is completed.
- Shortcut creation failures are explicit.
- Locked layouts cannot be silently overwritten by autosave.
- Variant deletion removes DeskRealm layout metadata only; it never deletes Desktop files or icons.

## Upgrade notes

Config version is now `10`.

Existing installs migrate automatically. Existing customized hotkeys are preserved. Only configs still using untouched legacy default hotkeys are moved to the new `Win+Shift+X/C/B/N` defaults.

New lock dictionaries are initialized empty until the user locks a layout, realm or variant from the UI:

```json
{
  "lockedIconLayouts": {},
  "lockedRealms": {},
  "lockedIconLayoutVariants": {}
}
```

## Recommended validation

After installing the release, test:

1. fresh first-run onboarding;
2. opening and hiding the UI from the tray;
3. hotkey capture and duplicate rejection;
4. disabling **Enable realm switching automation** and confirming hotkeys are ignored;
5. layout/realm/variant locks;
6. deleting a stale layout variant;
7. icon layout restore across the normal monitor setup and one alternate display topology;
8. compiled `.exe` icon and tray icon both show the DeskRealm `DR` logo.
